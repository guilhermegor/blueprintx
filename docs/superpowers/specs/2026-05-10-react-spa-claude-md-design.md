# Design: CLAUDE.md for react-spa-webpack scaffolded projects

**Date:** 2026-05-10  
**Status:** Approved  
**Skeleton:** `react-spa-webpack`

---

## Problem

Scaffolded React SPA projects have no AI guidance. When a developer opens one
with Claude Code, there are no guardrails on layer imports, factory usage, or
state management patterns — leading to layer violations and ad-hoc decisions
that diverge from the BlueprintX architecture.

## Goal

Ship a `CLAUDE.md` into every project generated by `bin/scaffold/ts_react_app.sh`
that gives Claude Code precise, variant-aware guardrails for the
react-spa-webpack architecture.

---

## Placement decision

Each TypeScript skeleton gets its own `CLAUDE.md` template. The React SPA
patterns (capabilities, Context/Zustand/RTK, factories, ports) are too specific
to share with a future Node API or library skeleton.

File location: `templates/react-spa-webpack/CLAUDE.md`

---

## Variant-aware mechanism

The scaffold already runs `envsubst` for `package.json`. We extend this to
`CLAUDE.md` using three new variables:

| Variable | React Context | Zustand | Redux Toolkit |
|---|---|---|---|
| `${STATE_MANAGEMENT_VARIANT}` | `React Context` | `Zustand` | `Redux Toolkit` |
| `${STATE_MANAGEMENT_DESC}` | use-cases.ts exports one custom hook per use-case (useState + useCallback). Each hook owns its loading, error, and result state. | use-cases.ts is a Zustand store (create<Store>()). State and async actions are co-located; the store is a singleton not tied to the React tree. | use-cases.ts is an RTK slice with createAsyncThunk actions. initialState, reducers, and thunks are co-located in one file. |
| `${STATE_MANAGEMENT_ANTIPATTERN}` | Don't lift hook state into a shared module — each hook is intentionally isolated. | Don't create more than one Zustand store per capability — merge new actions into the existing store. | Don't dispatch actions outside of thunks or hooks — keep all side effects inside the RTK layer. |

Variables are set in `apply_file_variants()` (already selects the state
management variant) and exported in `copy_common_templates()` alongside the
existing `PROJECT_NAME` / `PROJECT_DESCRIPTION` export.

An explicit variable list is passed to `envsubst` to prevent accidentally
expanding `${...}` inside code-fence blocks in the template.

---

## CLAUDE.md structure

1. **What this project is** — one-liner naming the project and state variant
2. **Commands** — dev / build / type-check / lint
3. **Architecture** — ASCII capability tree with per-layer annotations
4. **Layer import rules** — table: may import / must never import
5. **State management: `${STATE_MANAGEMENT_VARIANT}`** — variant description + anti-pattern
6. **Adding a new capability** — ordered steps enforcing domain-first discipline
7. **`factories.ts` rules** — correct vs. wrong code snippets
8. **Do / Don't** — quick-scan table of common mistakes
9. **Testing** — layer-per-layer strategy

## Changes to `bin/scaffold/ts_react_app.sh`

### `apply_file_variants()` — add variable assignments per branch

```bash
case "$STATE_MGMT_CHOICE" in
  2)
    STATE_MANAGEMENT_VARIANT="Zustand"
    STATE_MANAGEMENT_DESC="..."
    STATE_MANAGEMENT_ANTIPATTERN="..."
    mv ...
    ;;
  3)
    STATE_MANAGEMENT_VARIANT="Redux Toolkit"
    STATE_MANAGEMENT_DESC="..."
    STATE_MANAGEMENT_ANTIPATTERN="..."
    mv ...
    ;;
  *)
    STATE_MANAGEMENT_VARIANT="React Context"
    STATE_MANAGEMENT_DESC="..."
    STATE_MANAGEMENT_ANTIPATTERN="..."
    rm -f ...
    ;;
esac
```

### `copy_common_templates()` — export + envsubst

```bash
export PROJECT_NAME PROJECT_DESCRIPTION \
       STATE_MANAGEMENT_VARIANT STATE_MANAGEMENT_DESC STATE_MANAGEMENT_ANTIPATTERN

envsubst '${PROJECT_NAME} ${STATE_MANAGEMENT_VARIANT} ${STATE_MANAGEMENT_DESC} ${STATE_MANAGEMENT_ANTIPATTERN}' \
    < "$SKELETON_TEMPLATE_ROOT/CLAUDE.md" \
    > "$project_path/CLAUDE.md"
```
