# Service tag-driven changelog — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Give the four service skeletons (`mvc-service-native-db`, `mvc-service-orm-db`, `ddd-service-native-db`, `ddd-service-orm-db`) the same generated, tag-driven `docs/changelog.md` that `lib-minimal` has, mirroring the lib's online/offline split.

**Architecture:** Services gain git tags — offline via `make bump_version` → `cz bump`; online via a new `release.yaml` (`workflow_dispatch`) that tags + cuts a GitHub Release (no PyPI). The changelog page switches to the `--8<-- "CHANGELOG.md"` include, regenerated by `cz changelog` in the docs build. `pyproject.toml` stays the nominal version home. All shared edits land in `templates/python-common/` (single source), with `lib-minimal`'s scaffold stripping the service-only `release.yaml`.

**Tech Stack:** Make + bash scaffolding, commitizen (`cz`), Poetry, MkDocs (material + pymdownx.snippets), GitHub Actions.

## Global Constraints

- BlueprintX is a scaffolding tool: template files are copied **verbatim** into generated projects. `templates/python-common/` is the single source of truth for shared Python tooling; changes there propagate to all Python tiers on the next scaffold.
- `pyproject.toml` templates use `${VARIABLE}` placeholders resolved by `envsubst` — never replace with literals.
- Never call a bare `poetry` in a recipe/hook/`bin/*.sh` — route through `POETRY := bash bin/poetry_exec.sh` (Makefile) / `poetry_exec()` (tasks.sh) / `bash bin/poetry_exec.sh run …` (pre-commit).
- `Makefile` ↔ `tasks.sh` must stay in sync: every target has a matching function + case branch + help entry.
- Type-prefix naming, one class per file, NumPy docstrings, ruff line-length 99 tab-indent — for any Python touched.
- `pyproject.version` online for services stays **nominal** (changelog reads tags; nothing reads the field post-#41). `ponytail:` acknowledged simplification.
- `release.yaml` for services is GitHub-only — it must never ship in offline scaffolds (already gated by `apply_offline_mode` copying only from `templates/common/`; `copy_github_assets` copies `.github/` only when a remote is set up).
- Run `act` (via the `act` skill) on any changed/new workflow before pushing (act-first lesson).
- Do not commit or push without explicit user request.

**Verification model:** these are templates, so a "test" is: scaffold to a temp dir with `make dev` (or `make dev-clean`) and assert the generated file has the expected property (grep), plus `act` for CI and `shellcheck`/`bash -n` for shell. `make dev` scaffolds interactively — set the env vars the scaffold reads (`GITHUB_USERNAME`) and pipe menu answers, or run the scaffold script directly: `bash bin/scaffold/python_mvc_service.sh <name> <target_dir>` (inspect the script's `validate_inputs` for the exact arg order before running).

- [ ] **Step 1: Rewrite the `bump_version` recipe in `templates/python-common/Makefile`**

Replace the current recipe (`poetry version $(LEVEL)` + `git add`) with a `cz bump`. `cz bump` computes the next version from Conventional Commits, writes `pyproject.toml`, regenerates `CHANGELOG.md` (via `update_changelog_on_bump`), commits `bump: X.Y.Z`, and creates the tag. `LEVEL` is dropped (cz derives the increment); keep an optional explicit-version override via `--increment`/positional is unnecessary — cz reads commits. Use `--no-verify`-equivalent by bypassing the commit-msg/pre-commit hooks the same way `bin/git_merge_to_main.sh` does. `cz` is invoked through `$(POETRY) run`:

```makefile
# Cut a version bump from Conventional Commits: cz computes the next semver, writes it to
# pyproject.toml, regenerates CHANGELOG.md, commits "bump: X.Y.Z", and creates the vX.Y.Z tag.
# --git-output-to-stderr keeps stdout clean; the commit bypasses protect-branch like the
# offline merge helper does. OFFLINE tier only — the online scaffold strips this target
# (releases are cut by the release.yaml workflow).
bump_version:
	@$(POETRY) run cz bump --yes --git-output-to-stderr
	@echo "Version bumped to $$($(POETRY) run cz version --project 2>/dev/null || $(POETRY) version -s)"
```

> Note: `cz bump` invokes `git commit`; if the local `protect-branch`/`no-commit-to-branch` pre-commit blocks it on `main`, document (in Task 6) that the offline bump runs on a feature branch before `make git_merge_to_main`, OR add `--no-verify` semantics. During Step 4 verification, confirm which is needed on a real scaffold and lock the recipe accordingly.

- [ ] **Step 1: Author the service release workflow**

Model it on `templates/lib-minimal/.github/workflows/release-pypi.yaml`'s `details` + `github_release` jobs, but **drop the PyPI build/publish/version-gate jobs**. Minimal shape:

```yaml
name: Release
# Cut a versioned release for this service: create the vX.Y.Z tag and a GitHub Release.
# Services are deployed, not published to a package index — there is no PyPI step. The tag is
# what the docs workflow's `cz changelog` groups the changelog by. main is protected; this runs
# server-side with the built-in GITHUB_TOKEN, so no direct push to main is needed.
on:
  workflow_dispatch:
    inputs:
      version:
        description: "Release version, e.g. 1.4.0 (no leading v)"
        required: true

permissions:
  contents: write

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout Code
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Create tag and GitHub Release
        env:
          GH_TOKEN: ${{ github.token }}
          VERSION: ${{ inputs.version }}
        run: |
          TAG="v${VERSION}"
          if git rev-parse "$TAG" >/dev/null 2>&1; then
            echo "Tag $TAG already exists"; exit 1
          fi
          git tag -a "$TAG" -m "Release $TAG"
          git push origin "$TAG"
          gh release create "$TAG" --title "$TAG" --generate-notes
```

(Verify exact action versions/token wiring against `release-pypi.yaml` and match its conventions; `--generate-notes` gives GitHub's auto notes — the docs-site changelog comes from `cz changelog`.)
